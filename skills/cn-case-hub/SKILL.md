# 中国案例检索与核验

## 分工

- 用本 Skill 处理案例、裁判观点、案号溯源和类案报告。
- 用可用的权威现行法检索能力处理法规、规章、司法解释、条约和具体法条；本公共包不捆绑第三方法规抓取代码。
- 在诉讼事项中把经核验的案例记录交给 `legal-os-litigation`；不要把检索结果直接当成最终法律意见。

## 工作流

1. **定义任务。** 确认法域为中国大陆，区分演示检索与真实事项。真实事项至少识别法律关系、程序阶段、代表立场、争议焦点和目标用途；缺少的信息标记为待确认，不补造。
2. **拆解检索要素。** 分别提取案由/法律关系、核心争点、关键事实、法条或司法术语、程序与地域、支持己方和反向结果。为每个争点同时设计正向、反向和限制性检索词。
3. **生成官方检索式。** 优先运行 `scripts/build_official_queries.py`，或按 `references/source-registry.md` 生成同等查询。检索式不得包含真实姓名、身份证号、联系方式或未脱敏的私有事实。
4. **按来源层级检索。** 依次查人民法院案例库、最高人民法院指导性案例与典型案例、最高人民法院公报、专业法院及地方法院官方站点、中国裁判文书网。来源被登录、验证码、维护或脚本限制阻断时，记录 `[待登录检索]` 或 `[来源受限]`，不得绕过访问控制。
5. **打开并核验原文。** 搜索结果摘要只作线索。实际打开官方页面，核对标题、来源类型、发布机关、入库编号/指导案例号、案号、法院、裁判日期、程序、裁判结果、裁判要点和原文链接。官方典型案例没有案号或法院时如实留空。
6. **检查当前适用背景。** 查找与争点相关的现行法、司法解释及后续官方材料；必要时调用可用的权威现行法检索能力。不得仅因案例仍能搜索到就断言其未被废止、改判或不受新法影响。无法完成状态核验时标记 `[状态待核]`。
7. **建立案例记录。** 按 `references/output-contract.md` 生成结构化记录。正式使用的案例必须至少达到 `verified-source`；只有元数据或搜索摘要的条目不得支持实体结论。
8. **评估相似度和方向。** 分别比较法律关系、争点、关键事实、程序/裁判层级和时间/法源背景，给出高/中/低及理由。标记案例为支持、反向或混合；不得把同案由自动视为高相似。
9. **交付检索报告。** 先给结论和最相关案例，再列反向案例、缺口和继续检索建议。每项实体陈述紧邻官方链接；明确访问日期和覆盖边界，不声称穷尽全部案例。

## 来源与验证规则

- 读取 `references/source-registry.md` 选择官方入口和来源等级。
- 搜索引擎、媒体、律所文章、公众号、数据库摘要只能提供线索，不能替代官方原文。
- 指导性案例、入库参考案例、典型案例、普通裁判文书必须分别标识；不得混称为“判例”。
- 匿名典型案例无案号、法院或全文时，标为 `official-summary`，不得虚构缺失字段。
- 裁判结果与裁判规则分开记录；部分胜诉不得概括为一方“完全胜诉”。
- 引用裁判文书时保留一审、二审、再审关系；无法核验后续程序时写明限制。
- 涉及具体案件材料时，只把必要的脱敏检索要素提交给外部搜索服务。

## 输出最低要求

每份结果至少包括：

1. 检索范围、关键词、来源和访问日期；
2. 最相关案例表：名称、来源类型、编号/案号、法院、裁判日期、争点、结果、相似度、方向、官方链接；
3. 每个重点案例的关键事实、裁判理由、用途与限制；
4. 反向或限制性案例；
5. 当前法源衔接及未完成核验；
6. `Verification Notes`：已核验事项、受限来源、未核验事项和时点。

## 硬性停止条件

- 找不到官方原文：降为 `lead-only`，不得作为正式依据。
- 官方来源要求登录：停止自动访问并标记待登录，不抓取 Cookie、不展示凭证、不绕过验证码。
- 案号、法院、日期或裁判结果相互冲突：并列记录冲突，停止下结论。
- 只有同名案例或二手摘要：不得认定为同一案件。
- 现行法与旧案例可能不一致：先核验现行法，再说明案例的历史或限制性价值。

## 工具

- `scripts/build_official_queries.py`：从脱敏争点生成免费官方来源检索式。
- `scripts/validate_case_records.py`：校验案例记录 JSON 的必填字段、官方域名、来源等级和验证状态。
- `references/source-registry.md`：官方来源优先级、访问限制和查询策略。
- `references/output-contract.md`：案例记录与类案报告字段规范。
